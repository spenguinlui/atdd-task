# E2E 測試完整性規範

> 避免「只驗證顯示、不驗證操作」的假通過。

## 核心原則

**E2E 測試 = 模擬真實使用者的完整操作流程。** 不是截圖確認 UI 長對了就算通過。

## 規則

### 1. 禁止只驗證顯示（Read-Only E2E）

每個涉及資料變更的場景，必須執行完整的 **操作 → 儲存 → 重新載入 → 驗證持久化** 循環。

```
❌ 只驗證顯示
  1. 打開編輯頁面
  2. 確認欄位出現 → PASSED

✅ 完整操作循環
  1. 打開編輯頁面
  2. 確認欄位出現
  3. 修改欄位值
  4. 點擊儲存
  5. 確認儲存成功（無錯誤）
  6. 重新載入頁面
  7. 確認修改後的值已持久化 → PASSED
```

### 2. 同功能多頁面 = 多個獨立測試

不可假設「A 頁面通過，B 頁面也會通過」。不同頁面可能有：
- 不同的 Controller
- 不同的 Repository（儲存路徑）
- 不同的 Form Object（表單驗證）
- 不同的 View Template（顯示邏輯）

```
❌
  AC1: 費率欄位顯示（兩個頁面）→ 只測了一個頁面

✅
  AC1a: 費率欄位顯示 — 單據帳戶管理頁面 → 完整操作循環
  AC1b: 費率欄位顯示 — 用電戶契約資訊頁面 → 完整操作循環
```

### 3. E2E 場景必須包含「負面路徑」

除了 Happy Path，必須測試：
- 空值提交 → 應顯示驗證錯誤（非 500 頁面）
- 非法值提交 → 應顯示友善錯誤訊息
- 權限不足 → 應正確拒絕

### 4. 表單提交後必須驗證兩件事

1. **即時回應**：儲存成功提示 / 錯誤訊息（非 500、非 translation missing）
2. **資料持久化**：重新載入頁面後值仍正確

## BA/SA 配合規範

### BA 驗收條件必須區分「查看」和「操作」

```
❌ 模糊動詞
  AC1: 費用開關選「是」時，顯示費率輸入欄位

✅ 明確動詞 + 完整循環
  AC1: 費用開關選「是」時，顯示費率輸入欄位，修改費率後儲存成功，重新載入頁面確認費率已更新
```

### SA 涉及多頁面時，必須列出各頁面的儲存路徑

不同頁面的 Repository/Controller 路徑不同，是常見的漏測根因。

```
❌ 只列 UI 檔案
  UI 頁面兩處：
    - _account_form.slim
    - _form.slim

✅ 列出完整的 UI + 儲存路徑
  頁面 A — 單據帳戶管理：
    View: _account_form.slim
    Save: multi_account_actor.rb#save_externalized_fee

  頁面 B — 用電戶契約資訊：
    View: consumer_contracts/_form.slim
    Save: consumer_contract/wrapper.rb（獨立儲存路徑！）
```

## 常見假通過模式

| 模式 | 為什麼是假的 | 正確做法 |
|------|------------|---------|
| 頁面載入沒報錯 → PASSED | 只證明 template 沒語法錯誤 | 必須操作 + 儲存 |
| 欄位顯示正確 → PASSED | 只證明 HTML render 對了 | 必須修改 + 儲存 + 驗證持久化 |
| 用 JS 設值 + JS click → PASSED | 繞過 HTML5 validation 和 Turbolinks | 必須用滑鼠操作模擬真實使用者 |
| 一個頁面通過 → 另一個也標 PASSED | 不同頁面可能走不同儲存路徑 | 每個頁面獨立測試 |
| 提交成功沒報錯 → PASSED | 可能靜默失敗（值沒更新） | 重新載入驗證持久化 |
