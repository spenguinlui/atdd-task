Maintainer's Knowledgebase

六張卡片：四平面架構、三類擴充入口、hook wiring、引擎委派安全模型、自驗基建與設計考量。

架構概覽 / Architecture
  • 四平面：Orchestrator → Commands（.claude/commands/）→ Agents（.claude/agents/）→ MCP 狀態
  • 7 個 agent：specist / tester / coder / style-reviewer / risk-reviewer / gatekeeper / curator
  • 核心設計：context 不靠對話傳遞，全部落 MCP state
  • → /continue 可從任意 session、任意引擎恢復
  • config 集中在 .claude/config/（引擎登記 / 預算 / 工具安全 / 信心度）
4 平面 → 細節
怎麼擴充 / Extend
  • 加 Command：.claude/commands/<name>.md + 共用邏輯抽 shared/
  • 加 / 改 Agent：編 .claude/agents/<name>.md + agent-engines.yml 補一列
  • 加 Project：照 projects.yml.example 加 id + 路徑
  • 調預算：全局 budget.yml；單任務 JSON 的 budget 欄覆寫
  • 新交付物要驗收 → hooks/lib/validate_deliverables.py 補規則
5 入口 → 步驟
Wiring & Hooks
  • Hook 以「事件 + matcher」掛在 .claude/settings.json
  • 10 支 hook：信心度硬阻擋、交付物完整性、E2E 決策、spec 格式、metrics 記錄…
  • 主體 ≤100 行：讀 stdin、基本判斷、呼叫 lib/（Python 重邏輯可獨立測試）
  • lib/hooklog.sh 寫 .hook-log.jsonl 供觀測
  • 加 hook：寫主體 → settings.json 綁事件 → 末段呼叫 hooklog
10 hook → 綁定表
引擎委派 / Engine Dispatch
  • 同一份 agent prompt 可委派不同引擎：engine: claude（預設）或 codex（GPT）
  • shared/agent-dispatch.md 每次呼叫前查 agent-engines.yml
  • Plane-2（in-agent hook）委派 codex 後靜默失效
  • → 必在 Plane-1（orchestrator）補等效檢查，否則是漏洞
  • specist / curator 有人在環內互動，不適合委派 headless
Plane-1 / Plane-2 → 安全模型
自驗基建 / Self-verify
  • bash experiments/atdd-eval/run-self-verify.sh — 跑所有 scorer 的單一入口
  • 任一 scorer 失敗 → Stop hook exit 2 物理擋住 session 結束
  • 四 Pattern（寫新 scorer 挑一個套）：A 真實來源+drift · B 觸發+斷言 · C scorer+METRICS · D 快照+diff
  • 進度：7 scorer / 58 check / 47% mechanism 覆蓋（7/15）
  • 加 scorer：寫 test-<name>.sh（檔頭標 pattern）→ chmod +x → 更新 coverage.json
7 scorer · 58 check → 加 scorer
設計考量 & 驗證改動 / Rationale
  • 為什麼用 MCP state 而非對話記憶：可跨 session / 跨引擎恢復
  • 為什麼信心度是硬阻擋：軟提醒會被 AI 忽略
  • 為什麼 hook 主體 ≤100 行：業務邏輯不該滲入 hook 層
  • 為什麼外層 eval 獨立於 inner gatekeeper：一個問「這張任務好不好」，一個問「框架整體升還是降」
  • 改動驗收：預算天花板 / hook 大小 / 引擎委派 / 自驗全綠（見驗證表）
