每個階段的交付，
先驗收再前進

ATDD task 的核心是逐階段驗收交付產物：需求 → 規格 → 測試 → 實作 → 審查， 每一份 deliverable 都要先通過該階段的驗收閘門，才進入下一步——用層層把關換取穩定、可預期的開發。
把需求交給 AI，你只需在關鍵點確認；不用懂技術細節。

Browse Help Topics

六個入口卡片，從第一次設定環境到守門員驗收。卡住了用上面的搜尋框過濾。

快速開始 / Getting Started
  • 複製 .env.example 為 .env，填專案路徑與測試帳號
  • 啟動第一個任務：/feature sf_project, 專案審核流程
  • AI 會馬上開始問問題，直到對需求有 95% 把握
  • 逐階段推進：/continue；隨時查進度：/status
  • 不確定哪個變數怎麼填？問框架管理者
4 步上手 → 操作手冊
任務類型 / Task Types
  • /atdd {一段話需求} — 自然語言入口，AI 自動判斷類型
  • /feature — 開發新功能（完整流程）
  • /fix — 修 Bug（跳過規格階段）
  • /refactor — 重構（審查強調行為不變）
  • /epic — 大型功能，AI 先提拆分方案
5 種入口 前門
工作流程：AI 做什麼，你做什麼
  • 需求釐清 → 規格 → 測試 → 實作 → 審查 → 守門裁決
  • AI 在大部分階段自動跑，你只在交棒點確認
  • 人工介入只有六個時機：啟動任務 · 回答需求 · E2E 決策 · 確認規格 · 決定修復範圍 · GO 後結案
  • 每個 /continue 前都能先看上一階段產出
  • 「信心度」未達 95% 前 AI 不會開始實作
6 交棒點 → 流程詳解
指令速查 / Commands
  • 生命週期：/continue · /status · /done · /abort
  • 審查後修復：/fix-critical · /fix-high · /fix-all
  • E2E 測試：/test-run · /test-pause · /test-resume
  • 部署收尾：/deploy · /verify · /escape
  • 忘了指令？打 /guide 看速查表
40 指令 → 完整清單
產出與驗收 / Outputs & Acceptance
  • 驗收規格（Given / When / Then）— 確認 AI 理解你的需求
  • 自動化測試 + 已通過審查的程式碼變更
  • 人工驗收指南：守門員給的逐步操作步驟
  • 確認做對了：守門員 GO → 照驗收指南打勾 → /verify 完成
  • 守門員 NO-GO 會說明原因，依指示調整後再 /continue
GO / NO-GO 守門員
邊界 & 出錯怎麼辦 / Limits
  • 不做：直接動 production 資料、需求沒確認就實作、改你沒要求的部分
  • 需求越模糊問得越多 → 啟動前先想清楚主要業務規則
  • E2E 需 Chrome 開著且能連 staging、帳號有測試資料
  • /continue 沒動作 → 看 AI 提示補完上一階段
  • 任務卡死 → /abort 重開，或查操作手冊
