Architecture Overview
  • Four planes: Orchestrator → Commands (.claude/commands/) → Agents (.claude/agents/) → MCP state
  • 7 agents: specist / tester / coder / style-reviewer / risk-reviewer / gatekeeper / curator
  • Core design: context is never passed via conversation — it all lands in MCP state
  • → /continue can resume from any session, any engine
  • Config centralized in .claude/config/ (engines / budget / tool safety / confidence)
4 planes → Details
How to Extend
  • Add a Command: .claude/commands/<name>.md + shared logic into shared/
  • Add / change an Agent: edit .claude/agents/<name>.md + add a row in agent-engines.yml
  • Add a Project: follow projects.yml.example — id + path
  • Tune budget: global budget.yml; per-task override via the JSON budget field
  • New deliverable to verify → add a rule in hooks/lib/validate_deliverables.py
5 entries → Steps
Wiring & Hooks
  • Hooks bind by "event + matcher" in .claude/settings.json
  • 10 hooks: confidence hard-block, deliverable completeness, E2E decision, spec format, metrics recording…
  • Body ≤100 lines: read stdin, basic checks, call lib/ (Python heavy logic, independently testable)
  • lib/hooklog.sh writes .hook-log.jsonl for observability
  • Add a hook: write body → bind event in settings.json → call hooklog at the end
10 hooks → Binding table
Engine Dispatch
  • One agent prompt, multiple engines: engine: claude (default) or codex (GPT)
  • shared/agent-dispatch.md checks agent-engines.yml before each call
  • Plane-2 (in-agent hooks) silently goes dark once delegated to codex
  • → you must add equivalent checks in Plane-1 (orchestrator), or it's a hole
  • specist / curator have a human in the loop — not for headless delegation
Plane-1 / Plane-2 → Safety model
Self-verify Infrastructure
  • bash experiments/atdd-eval/run-self-verify.sh — single entry point for all scorers
  • Any scorer fails → Stop hook exit 2 physically blocks the session from ending
  • Four patterns (pick one for a new scorer): A source+drift · B trigger+assert · C scorer+METRICS · D snapshot+diff
  • Progress: 7 scorers / 58 checks / 47% mechanism coverage (7/15)
  • Add a scorer: write test-<name>.sh (tag the pattern in the header) → chmod +x → update coverage.json
7 scorers · 58 checks → Add a scorer
Design Rationale & Verifying Changes
  • Why MCP state over conversation memory: resumable across sessions and engines
  • Why confidence is a hard block: soft reminders get ignored by the AI
  • Why hook bodies stay ≤100 lines: business logic shouldn't seep into the hook layer
  • Why outer eval is separate from the inner gatekeeper: one asks "is this task good", the other "did the framework get better or worse"
  • Change acceptance: budget ceiling / hook size / engine dispatch / all scorers green (see the verify table)
